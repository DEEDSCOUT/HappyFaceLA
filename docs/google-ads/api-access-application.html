Business Model

Happy Faces LA is a solo-operated face painting and event entertainment business serving the Greater Los Angeles area. The company advertises its own services (face painting, balloon twisting, glitter tattoos, face gems) exclusively on its own Google Ads account (469-912-0105). It does not manage advertising for any other business, does not resell advertising services, and operates no third-party client accounts.

All Google Ads activity is owned and operated by the same individual who owns the business. The developer token lives in the Manager (MCC) account 634-051-0052 (Happy Face LA), which manages only the single client account above.

Tool Access / Use

The tool is a private, locally-executed MCP (Model Context Protocol) server built in Python. It runs exclusively on the business owner's personal computer and is never deployed to the internet, hosted on any server, or shared with any third-party user or organization.

Access is restricted to the owner's local machine only. Credentials are stored in a gitignored .env.local file and are never transmitted to or stored by any external service other than Google's own OAuth 2.0 infrastructure.

The tool integrates with VS Code + GitHub Copilot via the MCP protocol, allowing the owner to query and manage their Google Ads account through natural-language prompts inside their code editor. All API calls originate from the owner's local machine.

There is no web UI, no SaaS layer, no data warehouse, and no third-party recipients of any data returned by the API.

Tool Design

The tool consists of a single Python process (tools/google_ads_mcp/server.py) exposing named tools over the MCP stdio transport. When VS Code + Copilot invokes a tool, the server makes the corresponding Google Ads API call and returns structured JSON to the AI assistant, which then presents a human-readable summary to the owner.

Read operations (13 tools)

  • Pull account-level settings, currency, time zone, conversion tracking configuration
  • List all campaigns with status, channel type, budget, bid strategy, optimization score
  • Full diagnostics for a named campaign: asset groups, signals, locations, conversion goals, tracking health
  • List all conversion actions and customer/campaign-level conversion goals
  • List Performance Max asset groups and all creative assets (text, image, video)
  • List geo-targeting locations per campaign
  • List all keywords and campaign-level negative keywords
  • Pull last 30 days of change history
  • Generate and write a full Markdown diagnostics report to disk
  • Execute arbitrary read-only GAQL SELECT queries (DML keywords blocked)

Write operations (13 tools — all safety-gated)

Every write tool passes through a safety framework that enforces the following before any mutation is sent to Google:

  • Explicit approval token required for live mutations
  • Default mode is validate_only=true — Google validates the operation server-side without applying it
  • Before-snapshot saved to disk before every mutation
  • JSON mutation plan generated and saved before execution
  • After-snapshot and JSONL audit log entry written on completion
  • Budget changes >25% blocked unless an explicit override flag is passed
  • Campaign removal requires the word "REMOVE" in the reason string
  • New Search campaigns always created in PAUSED status
  • BROAD match keywords require an explicit "BROAD_OK" acknowledgment in reason

Write capabilities include: campaign status, daily budget, bid strategy, conversion action settings, customer and campaign conversion goals, campaign negative keywords, Search campaign creation, Search keyword management, Performance Max asset updates, geo-targeting, image asset upload, and apply-recommendation (hard-gated, cannot run in validate-only mode).

Tool Mockup — Typical Workflow

The tool has no graphical user interface. The owner types natural-language requests in VS Code Copilot chat; the AI translates these to tool invocations; results are returned as structured text in the editor. Below are representative interactions:

Example 1 — Read: Campaign Diagnostics

Owner: "Run diagnostics on my Kids Party campaign" Copilot invokes: google_ads_campaign_diagnostics("Kids Party Face Painting") Response summary: Campaign: Kids Party Face Painting Status: ENABLED | Channel: PERFORMANCE_MAX | Budget: $30.00/day Asset groups: 1 (ENABLED) Final URLs: ["https://happyfacesla.com/kids-birthday-party..."] Conversion goal: SUBMIT_LEAD_FORM — biddable: true ✅ Conversion action: generate_lead — primary_for_goal: true ✅ Diagnosis: No tracking misconfigurations detected.

Example 2 — Write: Budget Update (validate-only first)

Owner: "Increase my Kids Party budget to $40 — validate first" Copilot invokes: google_ads_update_campaign_budget({ "campaign_name_or_id": "Kids Party Face Painting", "new_daily_budget_usd": 40.00, "reason": "Testing higher budget for peak season", "validate_only": true }) Response: validate_only: true Plan saved: docs/google-ads/mutation-plans/2026-05-20T...budget.json Old budget: $30.00 → New budget: $40.00 (+33.3%) Google validation: PASSED (no errors) Owner: "Looks good — apply it" Copilot invokes same tool with validate_only: false, approval_token: "I_APPROVE_..." Response: Mutation applied. Audit entry written. After-snapshot saved.

Example 3 — Read: Diagnostics Report

Owner: "Generate my full Google Ads diagnostics report" Copilot invokes: google_ads_generate_diagnostics_report() Response: Written: docs/google-ads/diagnostics.md (12,840 bytes) Sections: Account · Campaigns · Conversion Actions · Goals · Assets · Locations · Change History
